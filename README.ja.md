## 🎯 主要な設計判断

### なぜキャラクターのコンポーネントではなく、独立したPartyManagerアクターにしたか

パーティシステムの生命周期は、特定のキャラクターの生命周期と結合してはいけません。フォーメーションマネージャーがCharacterのコンポーネントとして存在すると、そのキャラクターが死亡または交代した瞬間にシステムが破綻します。`APartyManager`を独立アクターとして抽出することで、リーダー交代やメンバーの死亡が「単純な状態変更」で済み、「システムの再構築」を必要としません。

このパターンは将来の敵モブシステムにも適用可能です。同じ`AGroupManager`の骨格に異なるフォーメーションコンポーネントを組み合わせることで、味方AIと敵AIでインフラを共有できます。

### なぜLine TraceではなくSphere Sweepで壁検知を行うか

Line Traceは厚みが0であるため、キャラクターのカプセルが実際には通過できない狭い隙間を「通過可能」と誤判定します。レイは通っても、キャラクター本体は壁にぶつかります。Sphere Sweepでキャラクターのカプセル半径と同等の判定を行うことで、実際の移動フットプリントと一致した検知が可能になり、壁スライディング補正が正しく発動します。

### なぜフォーメーション全体の縮小ではなく、VectorPlaneProjectによるスライディング補正を選んだか

壁にぶつかった単一スロットのために隊形全体を縮小すると、すべての仲間が一つの障害物に「機械的に縛られた」ように見えます。代わりに、ぶつかったスロットだけを壁面法線で`VectorPlaneProject`して投影することで、影響を受けたキャラクターだけが壁に沿って自然にスライディングし、他の仲間はV字隊形を維持します。視覚的な乱れを「実際に発生した場所」に限定する設計です。

### なぜ「モダンに見える」非同期LineTraceを採用しなかったか

仲間3人 + 距離ベースポーリング(50cm閾値)の構成では、空間クエリのコストは無視できる水準です(60fpsで毎秒約360回)。非同期化はコールバックの複雑性、ステイルデータ処理、チケットベースの防御コードを導入する一方、測定可能なパフォーマンス改善はありません。エージェント数30以上の大規模群衆シナリオなど、プロファイラで実際のボトルネックが確認された時点で再検討します。

## 🧩 検討中の課題 (Open Problems)

開発中に観察された実際のゲームフィール上の課題。各項目は「なぜその問題が起こるか」「各解決策のトレードオフは何か」を記録しています。実装の前に「理解」を重視するアプローチです。

### 1. プレイヤーに押された仲間が崖から落ちる

『エンドフィールド』では、プレイヤーが仲間を衝突で押した場合、仲間は歩行可能エリアにのみ押されます。崖から落ちません。一方、デフォルトの物理ベース押下にはこの制約がありません。

- **仮説**: 押された位置を毎フレーム最寄りのNavMesh点にクランプ、または仲間がエッジ近接閾値内にいる時に外向きの力を加える
- **方向性**: NavMeshエッジ検知 + エッジ接線方向への力ベクトル投影

### 2. 仲間が空中でジャンプしてしまう

仲間がリーダーのジャンプ入力を盲目的に模倣すると、平地でも何もない空中をジャンプしてしまいます。実際のゲームでは「経路が実際にジャンプを要求する時のみ」発動します。

- **仮説**: 設計者が配置するNavLinkマーキングと、プログラム的なギャップ検知のためのパス先読みを組み合わせる
- **方向性**: NavLink統合をメインに、未マーキング地形のフォールバックとしてパス先読み

### 3. 似た目標を持つエージェント間でRVOが「擦り合い」を起こす

急な方向転換やフォーメーション切替後、複数の仲間が近接スロットを占有しようとすると、RVOがクリーンな経路解決ではなく振動的な横歩きを引き起こします。RVOはリアクティブであり、プレディクティブではない — グループ認識が欠如しています。

- **仮説**: Detour Crowd Manager (UE5ネイティブ) がグループ文脈を持つ予測ベース回避を提供。フォーメーション遷移時のHungarianアルゴリズムによる最適スロット再割り当てが総変位を最小化
- **方向性**: Detour Crowdをベースレイヤーとし、フォーメーション遷移時にHungarianマッチングを適用

### 4. 仲間がプレイヤーの移動を妨げる

プレイヤーが仲間の経路に踏み込むと、両キャラクターが「擦り合う」状態でスタックします。ゴーストのように通り抜けさせるとゲームフィールが破綻し、RVOのチューニングには限界があります。

- **仮説**: Detour Crowdのグループ認識回避が役立ち、加えて仲間ごとの「譲歩 (yield)」行動 — 仲間が小範囲内のプレイヤー接近を検知して横にステップする
- **方向性**: Detour Crowd統合 + 仲間のBT/StateTreeに軽量な譲歩ロジック

## ✨ 実装済み機能

- **3レイヤーアーキテクチャ** (Manager / Controller / Character)、役割非依存のキャラクタークラス
- **数学ベースのフォーメーション配置**、足元基準の参照点 (Z軸浮遊バグの回避)
- **環境適応型の壁面スライディング** (Sphere Sweep + VectorPlaneProject)
- **NavMesh投影フォールバック**、到達不可能スロットへのリーダー方向引き戻し
- **バネベースのギャップスケーリング** (`FloatSpringInterp`) — リーダー速度連動の隊形伸縮
- **Quaternion遅延回転** — 仲間が回転に重く追従、機械的スナップを回避
- **距離 + 回転認識のキャッシュ無効化** — 必要な時のみ重いクエリを実行
