## ✨ 実装済み機能

- **3レイヤーアーキテクチャ** (Manager / Controller / Character)、役割非依存のキャラクタークラス
- **デザイナー駆動の隊形データ** (`UPrimaryDataAsset`) — 新隊形追加にコードビルド不要
- **NavMesh幅に基づくV ⇔ I自動切替** — Raycastで通路幅を測定、ヒステリシスで境界点滅防止
- **数学ベースのフォーメーション配置**、足元基準の参照点 (Z軸浮遊バグの回避)
- **環境認識のスロット補正** — 傾斜面Z補正、NavMesh第一検証、壁スライディングはフォールバック
- **バネベースのギャップスケーリング** (`FloatSpringInterp`) — リーダー速度連動の隊形伸縮
- **Quaternion遅延回転** — 仲間が回転に重く追従、機械的スナップを回避
- **距離 + 回転認識のキャッシュ無効化** — 必要な時のみ重いクエリを実行

## 🎯 主要な設計判断

### なぜキャラクターのコンポーネントではなく、独立したPartyManagerアクターにしたか

パーティシステムの生命周期は、特定のキャラクターの生命周期と結合してはいけません。フォーメーションマネージャーがCharacterのコンポーネントとして存在すると、そのキャラクターが死亡または交代した瞬間にシステムが破綻します。`APartyManager`を独立アクターとして抽出することで、リーダー交代やメンバーの死亡が「単純な状態変更」で済みます。同じ`AGroupManager`の骨格に異なるフォーメーションコンポーネントを組み合わせることで、味方AIと敵AIでインフラを共有できます。

### なぜLine TraceではなくSphere Sweepで壁検知を行うか

Line Traceは厚みが0であるため、キャラクターのカプセルが実際には通過できない狭い隙間を「通過可能」と誤判定します。レイは通っても、キャラクター本体は壁にぶつかります。Sphere Sweepでキャラクターのカプセル半径と同等の判定を行うことで、実際の移動フットプリントと一致した検知が可能になります。

### なぜ環境補正において「NavMeshが第一の真実」なのか

初期の流れは「壁検知 → スライディング → NavMesh投影」の順でしたが、スライディング結果がNavMesh外に着地してもそのまま採用されるため、傾斜地形の見た目をしているがNavMeshが無いエリアで仲間が到達不可能な座標を目指してしまう問題がありました。NavMesh投影を*第一の妥当性チェック*とし、壁スライディングは「投影失敗時の代替座標生成ツール」として再定義しました。各ヘルパーは単一責任を持ち、`AdjustLocationForEnvironment`はトップダウンの4段階オーケストレーションとして読めます。

### なぜ階層分離 — Managerが「モード」を決め、Componentが「モード内」を決めるのか

環境ベースのV/I切り替えは「平時モードの内部決定」なので、`FormationFollowComponent`内で完結させます。将来戦闘モードが導入される時、`APartyManager`は同じ決定を取り合うのではなく、アクティブなコンポーネント自体を差し替える役割になります。抽象化レベルを重ならせないことで、既存のコンポーネントを触らずに新しいモードを追加できる構造を確保しています。

<details>
<summary>その他の判断: ヒステリシス、非同期LineTraceの保留など</summary>

**境界点滅防止のためのヒステリシス**: 単一の幅閾値だと境界で隊形が点滅します。`NarrowThreshold=300`と`WideThreshold=500`の二重閾値を使い、反対側の閾値を完全に越えた時のみ切替が発火するようにしています。

**非同期LineTraceの保留**: 仲間3人 + 距離ベースポーリング(50cm閾値)の構成では、空間クエリのコストは無視できる水準です(60fpsで毎秒約360回)。非同期化はコールバックの複雑性とステイルデータ処理を導入する一方、測定可能な改善はありません。30エージェント超の大規模シナリオでプロファイラがボトルネックを示した時点で再検討します。

</details>

## 🧩 検討中の課題 (Open Problems)

開発中に観察された実際のゲームフィール上の課題。「なぜその問題が起こるか」「各解決策のトレードオフは何か」を記録しています。

### 1. プレイヤーに押された仲間が崖から落ちる

『エンドフィールド』では、プレイヤーが仲間を衝突で押した場合、仲間は歩行可能エリアにのみ押されます。崖から落ちません。デフォルトの物理ベース押下にはこの制約がありません。

- **方向性**: NavMeshエッジ検知 + エッジ接線方向への力ベクトル投影

### 2. 仲間が空中でジャンプしてしまう

仲間がリーダーのジャンプ入力を盲目的に模倣すると、平地でも何もない空中をジャンプしてしまいます。実際のゲームでは「経路が実際にジャンプを要求する時のみ」発動します。

- **方向性**: 設計者が配置するNavLinkマーキングをメインに、未マーキング地形のフォールバックとしてパス先読み

### 3. RVOではグループ的な混雑を解決できない

デフォルトのRVOはリアクティブで、グループ認識が欠如しています。急な方向転換や隊形切替後、複数の仲間が近接スロットを取り合い「擦り合い」が発生。プレイヤー進路上の仲間も同じ原因で発生します。

- **方向性**: Detour Crowd Manager (UE5ネイティブ)による予測ベース・グループ認識回避、加えて隊形遷移時のHungarianアルゴリズムによる最適スロット再割り当て
